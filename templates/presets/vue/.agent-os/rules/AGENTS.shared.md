# 项目 Agent 规则

- 主流程优先使用 `Compound Engineering`。
- 如果本机 Codex / Claude Code 未安装或无法读取 `Compound Engineering` 全局插件，不中断任务；按本文档的内置降级流程执行，并明确提示用户当前处于降级模式。
- 先判定任务类型，再决定流程与需加载的项目级 skills。
- 命中多个 skills 时必须组合加载，并在该任务后续步骤持续生效。
- 如果某个命中的项目级必需 skill 缺失或无法读取，必须立即中断，明确提示缺失 skill，并让用户选择如何继续。

## 任务分流

1. 新增功能、页面、模块开发
   - 先判断是否属于小改动。
   - 若需求不清晰，或不属于小改动，优先用 `ce:brainstorm`；不可用时按“内置降级流程”的需求澄清步骤执行。
   - 按命中规则加载项目级 skills。
   - 再优先进入 `ce:work`、`ce:review`、`ce:compound`；不可用时按“内置降级流程”的实现、自检、沉淀步骤执行。
2. `bugfix`、报错排查、小范围修复
   - 默认先走小改动判定。
   - 若问题范围不清晰，或不属于小改动，优先用 `ce:brainstorm`；不可用时按“内置降级流程”的问题澄清步骤执行。
   - 必载 `test-driven-development`，并按命中场景补测试/调试类 skills。
   - 再优先进入 `ce:work`、`ce:review`、`ce:compound`；不可用时按“内置降级流程”的实现、自检、沉淀步骤执行。

## 小改动判定

- 仅涉及代码文件。
- 仅统计新增或修改后的实际代码行数，不计删除。
- 新增/修改总计不超过 50 行。
- 改动目标明确，无额外需求澄清。
- 若涉及架构、跨模块、API/数据结构、状态管理、路由、权限、公共组件、共享基础设施、脚手架/构建配置，或验收标准不明确，则不能按小改动处理。

## 项目上下文初始化

- 首次处理非小改动前，先完成项目画像：阅读顶层目录、`package.json`、入口文件、路由配置、状态管理和核心模块结构。
- 对大型项目采用渐进式探索：先掌握顶层结构，再按任务相关模块深入；优先用搜索工具定向查找，避免一次性读取过多文件。
- 若形成稳定结论，应写入 `.agent-os/context/project-overview.md` 或项目已有的持久化文档，包含技术栈、目录结构、核心模块、关键约定和主要风险。
- 后续会话开始时，优先读取已有持久化上下文，避免重复分析和偏离既定架构。

## 项目级 skills 触发

- 设计稿 / MasterGo / 视觉还原 / 切图 / 设计转代码：`mastergo-to-code`
- UI/UX / 页面设计 / 视觉优化 / 交互体验 / 响应式 / 可访问性 / 表单表格图表体验：`ui-ux-pro-max`
- Vue 开发默认：`vue-best-practices`
- Vue 路由：`vue-router-best-practices`
- Vue 状态管理：`vue-pinia-best-practices`
- Vue JSX / TSX：`vue-jsx-best-practices`
- Vue Options API：`vue-options-api-best-practices`
- Vue 测试 / 回归验证：`vue-testing-best-practices`
- Vue 运行时错误 / 白屏 / 响应式异常 / 生命周期异常：`vue-debug-guides`
- 用户明确要求真实流程验证：结合 Playwright MCP

## 内置降级流程

当 `Compound Engineering` 不可用时，使用以下项目内置流程兜底：

1. 需求澄清 / 问题澄清
   - 先判断是否属于小改动。
   - 若不是小改动，先明确目标、范围、非目标、验收标准和主要风险。
   - 若存在多种实现方向，给出 2-3 个方案并说明推荐方案。
2. 实现
   - 读取相关文件和命中的项目级 skills。
   - 按现有代码风格小步修改。
   - 遵循最小变更原则，不做无关优化；除非用户明确要求或现有结构确实需要，否则不新增文件。
   - 如确需新增文件，先说明新增原因、路径和职责边界。
   - 涉及行为变化时优先补充或更新测试。
3. 自检
   - 优先运行与改动相关的测试、类型检查或构建命令。
   - 若无法运行验证，必须说明原因和剩余风险。
   - 以风险优先方式检查回归点、边界场景、遗漏测试、未使用变量、不必要依赖和敏感信息硬编码。
   - 前端改动需额外检查路由返回、菜单/权限硬编码、表单字段完整性、分页、加载态、空状态和响应式表现。
4. 沉淀
   - 如果本次问题形成了可复用经验，写入项目文档或对应 skill。
   - 若没有可复用经验，可以跳过沉淀，但需要在总结里说明。

## Spec / Task 约定

- 大功能必须先拆分为可独立验证的小任务。
- 需要持久化需求时，优先写入 `.agent-os/specs/<topic>/requirements.md`。
- 需要持久化任务拆分时，优先写入 `.agent-os/specs/<topic>/tasks.md`。
- 关键设计决策优先追加到 `.agent-os/context/decisions.md` 或项目已有决策文档。
- 每完成一个任务，都要更新任务状态并验证后再进入下一项。

## 输出与协作规则

- 项目文档、总结、提交信息默认使用中文，国际化项目或用户另有要求除外。
- 文档避免口语化 AI 痕迹，不使用“当然”“好的”“让我来”等开头。
- 完成后列出变更文件、验证结果和未覆盖风险。
- 团队项目建议将 `AGENTS.md`、`.agent-os/`、`.claude/`、`.codex/` 和同步脚本纳入 Git 管理；忽略模式仅适合个人临时增强。

## 补充

- 用户明确要求专项流程时，专项规则优先于默认小改动路径。
- 只要用户请求、文件路径、代码片段、报错信息、目录结构或上下文中出现命中条件，都视为触发。
- 具体专项细则以对应项目级 skill 的 `SKILL.md` 为准。
- `CLAUDE.md` 保持精简；共享规则统一写在这里，再通过 `pnpm agent-os:sync` 同步到根目录。
