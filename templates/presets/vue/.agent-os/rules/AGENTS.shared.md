# 项目 Agent 规则

- 主流程优先使用 `Compound Engineering`。
- 如果本机 Codex / Claude Code 未安装或无法读取 `Compound Engineering` 全局插件，不中断任务；按本文档的内置降级流程执行，并明确提示用户当前处于降级模式。
- 先判定任务类型，再决定流程与需加载的项目级 skills。
- 命中多个 skills 时必须组合加载，并在该任务后续步骤持续生效。
- 如果某个命中的项目级必需 skill 缺失或无法读取，必须立即中断，明确提示缺失 skill，并让用户选择如何继续。
- 不能只口头声明已进入某个流程或已加载某个 skill；必须实际读取并遵守对应 `SKILL.md` 的关键步骤。若因环境限制无法读取，必须说明并走降级流程。
- 本文件只定义通用工作流和强制闸门；前端、Vue、测试、调试等专项细节以下方命中的项目级 skill 为准。

## 强制工作闸门

以下闸门优先级高于普通实现步骤，未满足时不得直接写代码。

1. 需求与计划闸门
   - 非小改动、页面级功能、用户可见工作流、持久化、权限、数据结构相关需求，必须先明确：目标、非目标、数据是否持久化、是否影响后端、验收标准、主要风险。
   - 若用户描述中存在“所有都可以”“全部支持”“完整能力”“可编辑”“可配置”“支持保存”等范围不清词，必须先列出待确认点并请求确认，不得自行收敛范围。
   - 若 Agent 需要做产品语义决策，例如“仅本地临时生效，不写回数据源”，必须先获得用户确认。
   - 非小改动必须先给出可执行小任务清单，每个任务都要能独立验证。
   - 对页面级功能，计划至少包含：数据模型、交互入口、异常/空状态、验证方式、回退风险。
   - 需求分析必须先从用户目标和业务流程出发，再映射到现有系统实现，避免被当前代码结构带偏。
   - 计划确认前只允许阅读、搜索、梳理，不允许进行实现性文件修改。
2. 工作区闸门
   - 实现前必须执行并阅读 `git status --short`。
   - 若存在用户已有改动，必须在回复中明确哪些文件不是本次任务产生，并承诺不覆盖、不回滚。
   - 若目标文件已有未归属改动，必须先阅读 diff，确认能在其基础上修改；无法判断时询问用户。
3. 重构闸门
   - 默认优先局部增强现有代码，不删除、不整页重写。
   - 删除或整体替换既有页面、组件、API、路由、状态模块前，必须先说明原因、替代方案、风险和影响范围，并获得用户明确确认。
   - 对承载关键业务流程、用户可见结果或数据写入/流转的页面，未经确认禁止整页重构。
4. 验证闸门
   - 验证以 `Compound Engineering` 的流程步骤或内置降级流程的自检为准，Agent 负责完成工程侧基础验证，不默认代替人工业务验收。
   - 默认只验证本次改动文件及其直接影响面，例如相关类型、调用点、同目录测试、受影响组件；避免无差别运行全仓 ESLint、全仓类型检查或全量构建。
   - 优先运行面向改动范围的类型检查、ESLint、单元测试、组件测试或项目既有检查命令，确保本次改动代码健壮、类型定义正确、静态检查无新增错误。
   - 只有当改动涉及公共类型、全局配置、构建配置、路由/权限入口、状态管理基础设施、共享组件或跨模块契约时，才扩大到全局类型检查、全局 ESLint 或全量构建。
   - 涉及 UI、可视化结果、文件输出、表格、表单、交互编辑的任务，Agent 只需说明已完成的工程验证和未覆盖的真实业务路径；最终页面、登录态、权限、代理、弹窗入口和业务数据由人工验收。
   - 新增纯函数或工具文件只能用于隔离复杂数据转换；必须通过类型、测试或调用点检查证明其接入关系正确。
   - 若只能运行全局检查且全局检查存在历史错误，必须过滤本次改动文件和直接影响面，区分新增问题与既有问题。
   - 若项目缺少可运行的定向检查命令，或检查命令因环境/依赖阻塞无法运行，必须说明阻塞原因、已完成的替代检查和剩余风险。
   - 验证失败时不得用“历史错误很多”直接掩盖；必须区分本次改动相关错误与仓库既有错误。
5. 单次变更闸门
   - 大文件或大功能必须分批修改，每批完成后先验证再继续。
   - 单次改动避免超过可审查范围；如需要大范围调整，先拆分为多个任务。
   - 禁止把多个互不相关的需求混在一次实现中。

## 任务分流

1. 新增功能、页面、模块开发
   - 先判断是否属于小改动。
   - 若需求不清晰，或不属于小改动，先通过“需求与计划闸门”；优先用 `ce:brainstorm` / `ce:plan`，不可用时按“内置降级流程”的需求澄清和任务拆分步骤执行。
   - 按命中规则加载项目级 skills。
   - 再优先进入 `ce:work`、`ce:review`、`ce:compound`；不可用时按“内置降级流程”的实现、自检、沉淀步骤执行。
2. `bugfix`、报错排查、小范围修复
   - 默认先走小改动判定。
   - 若问题范围不清晰，或不属于小改动，优先用 `ce:brainstorm`；不可用时按“内置降级流程”的问题澄清步骤执行。
   - 必载 `test-driven-development`，并按命中场景补测试/调试类 skills。
   - 再优先进入 `ce:work`、`ce:review`、`ce:compound`；不可用时按“内置降级流程”的实现、自检、沉淀步骤执行。

## 小改动判定

- 仅涉及代码文件。
- 仅统计新增或修改后的实际代码行数，不计删除。
- 新增/修改总计不超过 50 行。
- 改动目标明确，无额外需求澄清。
- 若涉及架构、跨模块、API/数据结构、状态管理、路由、权限、公共组件、共享基础设施、脚手架/构建配置，或验收标准不明确，则不能按小改动处理。

## 项目上下文初始化

- 首次处理非小改动前，先完成项目画像：阅读顶层目录、`package.json`、入口文件、路由配置、状态管理和核心模块结构。
- 对大型项目采用渐进式探索：先掌握顶层结构，再按任务相关模块深入；优先用搜索工具定向查找，避免一次性读取过多文件。
- 若形成稳定结论，优先写入项目已有持久化文档；如果项目没有既定位置，可写入 `docs/agent-os/project-overview.md`。
- 后续会话开始时，优先读取已有持久化上下文，避免重复分析和偏离既定架构。

## 项目级 skills 触发

- 设计稿 / MasterGo / 视觉还原 / 切图 / 设计转代码：`mastergo-to-code`
- UI/UX / 页面设计 / 视觉优化 / 交互体验 / 响应式 / 可访问性 / 表单表格图表体验：`ui-ux-pro-max`
- Vue 开发默认：`vue-best-practices`
- Vue 路由：`vue-router-best-practices`
- Vue 状态管理：`vue-pinia-best-practices`
- Vue JSX / TSX：`vue-jsx-best-practices`
- Vue Options API：`vue-options-api-best-practices`
- Vue 测试 / 回归验证：`vue-testing-best-practices`
- Vue 运行时错误 / 白屏 / 响应式异常 / 生命周期异常：`vue-debug-guides`
- 用户明确要求真实流程验证：先确认登录态、代理、权限、入口和数据条件，再结合 Playwright MCP
- 具体专项规则不在本文件展开；命中后必须读取对应 `SKILL.md`。

## 内置降级流程

当 `Compound Engineering` 不可用时，使用以下项目内置流程兜底：

1. 需求澄清 / 问题澄清
   - 先判断是否属于小改动；若不是小改动，执行“需求与计划闸门”。
   - 若存在多种实现方向，给出 2-3 个方案并说明推荐方案。
2. 实现
   - 读取相关文件和命中的项目级 skills。
   - 先完成工作区闸门，再修改文件。
   - 按现有代码风格小步修改。
   - 遵循最小变更原则，不做无关优化；除非用户明确要求或现有结构确实需要，否则不新增文件。
   - 如确需新增文件，先说明新增原因、路径和职责边界。
   - 涉及行为变化时优先补充或更新测试。
   - 禁止为了方便实现而绕开或丢弃原有业务逻辑、样式、权限和既有用户可见行为。
3. 自检
   - 执行“验证闸门”，优先运行本次改动范围内的测试、类型检查或静态检查命令。
   - 以风险优先方式检查回归点、边界场景、遗漏测试、未使用变量、不必要依赖和敏感信息硬编码。
4. 沉淀
   - 按“沉淀规则”判断、写入并反馈。

## Spec / Task 约定

- 需要持久化规格或任务拆分时，沿用“需求与计划闸门”中已确认的目标、非目标、用户确认项、实现方案、验收标准和可独立验证的小任务，避免生成另一套口径。
- 需要持久化需求时，优先写入项目已有规格目录；如果项目没有既定位置，可写入 `docs/agent-os/specs/<topic>/requirements.md`。
- 需要持久化任务拆分时，优先写入项目已有任务文档；如果项目没有既定位置，可写入 `docs/agent-os/specs/<topic>/tasks.md`。
- 关键设计决策优先追加到项目已有决策文档；如果项目没有既定位置，可写入 `docs/agent-os/decisions.md`。
- 每完成一个任务，都要更新任务状态并验证后再进入下一项。

## 沉淀规则

沉淀是指把本次任务中可复用的经验、约束、排错路径或设计决策写入项目内可追踪位置，供后续 Agent 或团队成员复用。

1. 需要沉淀的情况
   - 发现新的项目约定、目录结构、业务流程、权限/路由/状态管理规则。
   - 解决了可复现的报错、构建问题、环境问题或兼容性问题。
   - 形成了可复用的实现模式、组件使用方式、接口数据转换方式或验证方法。
   - 用户明确要求记录、复盘、总结、沉淀。
2. 默认沉淀位置
   - 项目画像和上下文：优先写入项目已有文档；否则写入 `docs/agent-os/project-overview.md`。
   - 需求、任务、验收标准：优先写入项目已有规格目录；否则写入 `docs/agent-os/specs/<topic>/requirements.md` 和 `docs/agent-os/specs/<topic>/tasks.md`。
   - 关键设计决策：优先写入项目已有决策文档；否则写入 `docs/agent-os/decisions.md`。
   - 可复用问题解决方案：优先写入项目已有知识库；否则写入 `docs/agent-os/solutions/<topic>.md`。
   - 专项 skill 能力增强：仅当用户明确允许维护 skill 时，才更新对应 `skills/<skill-name>/`。
3. 沉淀内容要求
   - 必须包含：背景、适用场景、做法、验证方式、风险或不适用场景。
   - 不记录一次性聊天过程，不记录无复用价值的流水账。
   - 不把业务敏感数据、账号、密钥、真实客户隐私写入沉淀文档。
4. 完成反馈
   - 最终回复必须列出“沉淀结果”。
   - 若已沉淀，列出具体文件路径和新增/更新内容摘要。
   - 若未沉淀，写明“未沉淀”及原因。

## 命令与本地服务规则

- Windows / PowerShell 环境优先使用 `pnpm.cmd`、`npm.cmd`，避免直接调用可能被执行策略拦截的 `.ps1`。
- PowerShell 搜索命令中如包含复杂引号，优先拆成多条简单命令，避免单双引号嵌套导致误判。
- 若命令因沙箱、权限、依赖或端口问题无法运行，说明原因、替代检查和建议复现命令。

## 禁止模式

- 禁止在需求不清晰时自行决定保存策略、数据来源、交互范围或最终输出语义。
- 禁止未确认就删除并重写既有业务页面。
- 禁止在没有 PID/日志/端口归属的情况下留下后台服务。
- 禁止把“历史类型错误很多”作为跳过本次改动检查的理由；必须过滤本次改动文件确认。
- 禁止引入不必要的新依赖、新目录或新抽象。
- 禁止在未确认影响范围时修改菜单、权限、路由守卫、全局状态和构建配置。

## 输出与协作规则

- 项目文档、总结、提交信息默认使用中文，国际化项目或用户另有要求除外。
- 文档避免口语化 AI 痕迹，不使用“当然”“好的”“让我来”等开头。
- 完成后列出变更文件、验证结果、沉淀结果和未覆盖风险。
- 重要变更必须说明修改原因；无法验证的内容必须说明阻塞原因和建议复现命令。
- 除非用户明确要求，否则不主动提交 commit；需要提交时，提交信息使用中文并保持单一主题。
- 团队项目建议将项目采用的 Agent 规则文件和本地 skills 纳入 Git 管理；忽略模式仅适合个人临时增强。

## 补充

- 用户明确要求专项流程时，专项规则优先于默认小改动路径。
- 只要用户请求、文件路径、代码片段、报错信息、目录结构或上下文中出现命中条件，都视为触发。
- 具体专项细则以对应项目级 skill 的 `SKILL.md` 为准。
- 如果项目存在专门的 Agent 规则维护文档，以该文档为规则真源；否则以当前工具入口文件为准。
